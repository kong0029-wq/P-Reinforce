## 1. 모바일 구글 로그인 403 에러 해결 (`disallowed_useragent`)

### 🔍 문제 현상
사용자가 카카오톡, 네이버, 인스타그램 등 **모바일 인앱 브라우저**에서 구글 로그인을 시도할 때 `403: disallowed_useragent` 에러가 발생하며 로그인이 차단되는 현상이 발생했습니다.

### 🧩 원인 분석
*   **Google의 보안 정책**: Google은 보안상의 이유로 임베디드 프레임워크(WebView)에서의 OAuth 요청을 차단합니다.
*   **인앱 브라우저의 한계**: 카카오톡 등의 앱 내부에서 열리는 브라우저는 독립적인 시스템 브라우저가 아닌 WebView 형태이므로 Google의 차단 대상에 해당합니다.

### 💡 해결 방법 (3단계 방어 전략)
1.  **인앱 브라우저 정밀 감지**: `navigator.userAgent`를 분석하여 카카오톡, 네이버, 페이스북 등 10여 개의 주요 인앱 브라우저를 감지하는 로직을 구현했습니다.
2.  **시스템 브라우저 유도 모달**: 인앱 브라우저가 감지되면 "외부 브라우저(Chrome/Safari)로 열기"를 안내하는 전용 UI 모달을 띄우고, URL 복사 기능을 제공하여 사용자 이탈을 막았습니다.
3.  **패턴 최적화 (반복 수정)**: 초기에는 `wv`와 같은 짧은 패턴으로 인해 일반 Chrome에서도 모달이 뜨는 '오탐지'가 있었으나, `'; wv)'` 등 정밀한 패턴으로 수정하여 해결했습니다.

## 2. 로그인 흐름 안정화 및 역할 선택 기능 복구

### 🔍 문제 현상
모바일 로그인을 위해 '리다이렉트' 방식을 강화하는 과정에서, 어제 설정했던 **"신규 가입 시 일반/사업자 선택 모달"**이 나타나지 않고 바로 홈으로 이동해버리는 문제가 발생했습니다.

### 🧩 원인 분석
*   **경쟁 상태(Race Condition)**: Firebase의 `onAuthStateChanged`(로그인 상태 감시)가 `getRedirectResult`(리다이렉트 결과 처리)보다 먼저 실행되었습니다.
*   **자동 라우팅의 부작용**: 로그인이 감지되자마자 홈으로 보내버리는 로직이 신규 유저의 정보 입력 단계를 건너뛰게 만들었습니다.

### 💡 해결 방법
1.  **실행 순서 강제**: `redirectChecked`라는 상태 변수를 도입하여, 리다이렉트 결과 처리가 완전히 끝난 후에만 자동 라우팅 로직이 작동하도록 설계했습니다.
2.  **사용자 존재 여부 확인**: Firestore에 유저 문서가 있는 **기존 사용자**만 자동으로 홈으로 보내고, 문서가 없는 **신규 사용자**는 반드시 역할 선택 모달을 거치도록 로직을 분리했습니다.

## 3. 회원 탈퇴(계정 삭제) 기능 구현

### 🔍 구현 내용
사용자가 스스로 계정을 삭제할 수 있는 기능을 일반 사용자 대시보드와 사업자 설정 탭에 각각 추가했습니다.

### ⚙️ 상세 로직
1.  **2단계 확인 시스템**: 실수로 탈퇴하는 것을 방지하기 위해 "정말 탈퇴하시겠습니까?" → "최종 확인(복구 불가)"의 2중 컨펌 과정을 적용했습니다.
2.  **데이터 무결성 유지**: 단순히 Auth 계정만 지우는 것이 아니라, 사용자와 연결된 **예약 내역, 수업 정보(사업자), 로그인 로그, Firestore 문서**를 순차적으로 삭제하여 데이터 찌꺼기가 남지 않게 했습니다.
3.  **최근 로그인 예외 처리**: 보안상 로그온한 지 오래된 경우 탈퇴가 실패할 수 있는데, 이를 감지하여 "재로그인 후 시도"하라는 안내 메시지를 추가했습니다.
