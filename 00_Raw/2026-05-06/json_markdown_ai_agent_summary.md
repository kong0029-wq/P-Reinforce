# 1. JSON이란?

JSON은 정보를 컴퓨터가 읽기 쉽게 정리해서 적는 방법입니다.

아주 쉽게 말하면:

> JSON = 컴퓨터에게 주는 정리표

예를 들어 동물 정보를 JSON으로 쓰면 다음과 같습니다.

```json
{
  "이름": "초코",
  "나이": 2,
  "종류": "강아지"
}
```

이 뜻은 다음과 같습니다.

- 이름은 초코
- 나이는 2
- 종류는 강아지

JSON은 사람이 보기에도 어느 정도 읽기 쉽고, 컴퓨터가 처리하기에도 좋습니다.

---

# 2. JSON의 기본 기호

JSON에서 자주 쓰는 기호는 다음과 같습니다.

| 기호 | 뜻 | 예시 |
|---|---|---|
| `{ }` | 정보 묶음 | `{ "이름": "초코" }` |
| `" "` | 글자 표시 | `"초코"` |
| `:` | 이름과 값을 연결 | `"나이": 2` |
| `,` | 다음 정보와 구분 | `"이름": "초코", "나이": 2` |
| `[ ]` | 여러 개 목록 | `["산책", "간식"]` |

---

# 3. JSON 값의 종류

JSON에는 여러 종류의 값을 넣을 수 있습니다.

## 3.1 글자

글자는 큰따옴표로 감쌉니다.

```json
{
  "이름": "민수"
}
```

## 3.2 숫자

숫자는 큰따옴표 없이 씁니다.

```json
{
  "나이": 10
}
```

## 3.3 목록

여러 개의 값을 넣을 때는 `[ ]`를 씁니다.

```json
{
  "좋아하는 것": ["산책", "간식", "공놀이"]
}
```

## 3.4 안에 또 다른 정보 넣기

JSON 안에 JSON처럼 또 다른 정보 묶음을 넣을 수 있습니다.

```json
{
  "이름": "보리",
  "정보": {
    "나이": 4,
    "종류": "고양이"
  }
}
```

이건 큰 상자 안에 작은 상자가 들어간 것과 비슷합니다.

# 4. true, false, null

JSON에는 특별한 값 3개가 있습니다.

| JSON 값 | 쉬운 뜻 |
|---|---|
| `true` | 응, 맞아, 있어, 했어 |
| `false` | 아니, 없어, 안 했어 |
| `null` | 비어 있음, 아직 모름, 없음 |

예시:

```json
{
  "강아지니?": true,
  "배고프니?": false,
  "별명": null
}
```

뜻:

- 강아지가 맞음
- 배고프지 않음
- 별명은 아직 없음

중요한 점은 `true`, `false`, `null`에는 큰따옴표를 붙이지 않는다는 것입니다.

```json
{
  "배고프니?": false
}
```

이건 진짜 Boolean 값입니다.

하지만 아래처럼 쓰면 그냥 글자가 됩니다.

```json
{
  "배고프니?": "false"
}
```

# 5. JSON에서 자주 하는 실수

## 5.1 마지막 쉼표

틀린 예:

```json
{
  "이름": "민수",
  "나이": 10,
}
```

마지막 값 뒤에는 쉼표를 쓰지 않는 것이 안전합니다.

맞는 예:

```json
{
  "이름": "민수",
  "나이": 10
}
```

## 5.2 따옴표 빠뜨리기

틀린 예:

```json
{
  "잠자고 있니?: false,
  "메모": null
}
```

맞는 예:

```json
{
  "잠자고 있니?": false,
  "메모": null
}
```

항목 이름은 큰따옴표로 정확히 감싸야 합니다.

## 5.3 닫는 중괄호 빠뜨리기

틀린 예:

```json
{
  "이름": "나비",
  "나이": 1
```

맞는 예:

```json
{
  "이름": "나비",
  "나이": 1
}
```

# 6. JSON만 쓰는 걸까?

아닙니다. JSON 말고도 정보를 정리하는 형식은 여러 가지가 있습니다.

대표적으로 다음이 있습니다.

- JSON
- XML
- YAML
- CSV
- TOML

## 6.1 JSON

```json
{
  "이름": "초코",
  "나이": 2
}
```

특징:

- 웹, 앱, 서버에서 많이 사용
- 컴퓨터가 처리하기 좋음
- 규칙이 명확함

## 6.2 XML

```xml
<동물>
  <이름>초코</이름>
  <나이>2</나이>
</동물>
```

특징:

- 태그를 사용함
- 오래된 시스템이나 문서 구조에서 사용됨
- JSON보다 길어 보일 수 있음

## 6.3 YAML

```yaml
이름: 초코
나이: 2
좋아하는것:
  - 산책
  - 간식
```

특징:

- 사람이 읽기 편함
- 설정 파일에 자주 사용
- 들여쓰기에 민감함

## 6.4 CSV

```csv
이름,나이,종류
초코,2,강아지
보리,4,고양이
```

특징:

- 표 형태 데이터에 좋음
- 엑셀과 비슷함
- 복잡한 구조 표현은 어려움

## 6.5 TOML

```toml
이름 = "초코"
나이 = 2
```

특징:

- 설정 파일에 자주 사용
- 읽기 쉬운 편
- JSON만큼 범용적으로 쓰이지는 않음

# 7. Markdown이란?

Markdown은 사람이 읽기 좋은 문서를 쉽게 쓰기 위한 형식입니다.

아주 쉽게 말하면:

> Markdown = 사람이 보기 좋은 글쓰기 방법

예시:

```markdown
# 강아지 소개

- 이름: 초코
- 나이: 2
- 종류: 강아지
```

이건 제목과 목록이 있는 문서처럼 보입니다.

---

# 8. JSON과 Markdown의 차이

가장 중요한 차이는 목적입니다.

| 구분 | JSON | Markdown |
|---|---|---|
| 목적 | 데이터 정리 | 문서 작성 |
| 주로 읽는 대상 | 컴퓨터 | 사람 |
| 쓰는 곳 | 앱, 서버, API, 설정값 | 설명서, 메모, README, 프롬프트 |
| 특징 | 규칙이 엄격함 | 글처럼 쓰기 쉬움 |
| 느낌 | 정리표 | 문서/노트 |

한 줄로 정리하면:

> JSON은 컴퓨터에게 주는 정리표이고, Markdown은 사람에게 보여주는 예쁜 글입니다.

# 10. Markdown 기본 기호

Markdown에서 자주 쓰는 기호는 다음과 같습니다.

| 기호 | 뜻 | 예시 |
|---|---|---|
| `#` | 제목 | `# 제목` |
| `-` | 목록 | `- 사과` |
| `*` | 목록 또는 기울임 | `*강조*` |
| `1.` | 번호 목록 | `1. 첫 번째` |
| `** **` | 굵게 | `**중요**` |
| `_ _` | 기울임 | `_강조_` |
| `` ` ` `` | 코드처럼 표시 | `` `true` `` |
| `[ ]( )` | 링크 | `[네이버](https://naver.com)` |
| `>` | 인용문 | `> 좋은 말` |
| `---` | 구분선 | `---` |
| `|` | 표 | `| 이름 | 나이 |` |
| `- [ ]` | 체크박스 | `- [ ] 숙제하기` |
| `- [x]` | 완료 체크박스 | `- [x] 양치하기` |

# 13. AI에게 프롬프트를 줄 때 왜 Markdown을 많이 쓸까?

AI에게 주는 프롬프트는 꼭 Markdown이어야 하는 것은 아닙니다.

AI는 일반 문장도 읽고, JSON도 읽고, Markdown도 읽을 수 있습니다.

그런데 프롬프트에서는 Markdown이 많이 쓰입니다.

이유는 다음과 같습니다.

- 사람이 쓰기 편함
- 제목, 목록, 예시를 정리하기 좋음
- 긴 지시문을 문서처럼 관리하기 좋음
- AI도 구조를 파악하기 쉬움

예시:

```markdown
# 역할
너는 친절한 수학 선생님이다.

# 목표
- 초보자도 이해하게 설명한다
- 쉬운 예시를 든다

# 금지사항
- 어려운 말만 나열하지 않는다
- 사용자를 무시하지 않는다
```

이런 구조는 사람이 읽고 수정하기 쉽습니다.

# 14. 프롬프트에 JSON을 쓰면 언제 좋을까?

JSON은 입력값이 딱 정해져 있을 때 좋습니다.

예시:

```json
{
  "task": "product_summary",
  "language": "ko",
  "tone": "friendly",
  "max_length": 200,
  "product": {
    "name": "무선 마우스",
    "price": 29000
  }
}
```

이런 경우 JSON이 좋습니다.

- 프로그램이 자동으로 프롬프트를 만들 때
- 입력값을 정확히 전달해야 할 때
- 앱이나 서버에서 설정값처럼 관리할 때
- 여러 AI agent 설정을 저장할 때

# 15. Markdown과 JSON을 같이 쓰는 방식

실제로는 Markdown과 JSON을 같이 쓰는 경우도 많습니다.

```markdown
# 작업
아래 상품 정보를 바탕으로 소개문을 써줘.

# 상품 데이터
```json
{
  "이름": "무선 마우스",
  "가격": 29000,
  "색상": ["검정", "흰색"]
}
```

# 조건
- 친절한 말투
- 3문장으로 작성
```

이 방식은 두 가지 장점을 함께 씁니다.

- Markdown: 사람이 전체 지시를 읽기 좋음
- JSON: 데이터가 정확하게 정리됨

# 16. AI Agent에게 페르소나를 입힐 때는 무엇을 쓰면 좋을까?

AI agent에게 페르소나를 입힐 때는 보통 Markdown이 더 좋습니다.

이유:

- 역할, 말투, 성격, 규칙을 자연스럽게 설명할 수 있음
- 사람이 읽고 고치기 쉬움
- 예시와 금지사항을 함께 넣기 좋음

하지만 JSON도 필요할 수 있습니다.

정리하면:

| 상황 | 추천 형식 |
|---|---|
| 페르소나를 처음 설계할 때 | Markdown |
| 역할, 말투, 규칙을 설명할 때 | Markdown |
| 앱이나 서버에서 설정값으로 저장할 때 | JSON |
| 여러 agent 설정을 코드로 관리할 때 | JSON |
| 실전에서 가장 좋은 방식 | Markdown + JSON 조합 |

# 17. AI Agent 페르소나 Markdown 템플릿

```markdown
# 역할
너는 [어떤 역할인지] 담당하는 AI agent다.

# 목표
- [도와줘야 하는 목표 1]
- [도와줘야 하는 목표 2]

# 대상 사용자
- [초보자 / 전문가 / 고객 / 학생 등]

# 말투
- [친절하고 부드럽게]
- [짧고 명확하게]
- [전문적이지만 어렵지 않게]

# 성격
- [차분함]
- [친절함]
- [논리적임]
- [격려하는 스타일]

# 답변 방식
- 먼저 핵심부터 말한다
- 필요한 경우 예시를 든다
- 어려운 용어는 쉽게 풀어서 설명한다
- 마지막에 짧게 정리한다

# 반드시 지킬 규칙
- 모르면 모른다고 말한다
- 추측은 추측이라고 밝힌다
- 사용자를 비난하거나 무시하지 않는다
- 요청 범위를 벗어나면 먼저 범위를 분명히 한다

# 금지사항
- 지나치게 장황하게 말하지 않는다
- 근거 없는 확신을 하지 않는다
- 어려운 전문용어만 나열하지 않는다

# 출력 형식
- 기본적으로 짧은 문단으로 답변한다
- 필요하면 목록으로 정리한다
- 필요하면 예시를 포함한다
```

# 18. AI Agent 페르소나 Markdown 실제 예시

```markdown
# 역할
너는 초보자에게 어려운 개념을 쉽게 설명하는 AI 선생님이다.

# 목표
- 사용자가 처음 듣는 개념도 이해할 수 있게 돕는다
- 복잡한 내용을 쉬운 말로 바꿔 설명한다

# 대상 사용자
- 초보자
- 비전문가
- 학생

# 말투
- 친절하고 부드럽게 말한다
- 겁주지 않고 편안하게 설명한다
- 짧고 또렷하게 말한다

# 성격
- 차분하다
- 인내심이 많다
- 격려를 잘한다

# 답변 방식
- 먼저 한 줄로 쉽게 설명한다
- 다음에 비유를 든다
- 그 다음 예시를 보여준다
- 마지막에 핵심을 짧게 정리한다

# 반드시 지킬 규칙
- 어려운 용어를 쓰면 바로 풀어서 설명한다
- 사용자가 헷갈릴 만한 부분은 다시 짚어준다
- 모르는 내용은 아는 척하지 않는다

# 금지사항
- 잘난 척하는 말투 금지
- 너무 긴 문장 남발 금지
- 전문용어만 나열하는 설명 금지
```

# 21. 시스템/프로그램이란 무엇인가?

여기서 말하는 시스템이나 프로그램은 사람이 아니라 코드가 자동으로 읽고 처리하는 환경을 뜻합니다.

쉽게 말하면:

> 시스템/프로그램 = 앱, 서버, 코드, 자동화 도구처럼 사람이 직접 읽는 것이 아니라 컴퓨터가 읽고 처리하는 것

# 22. 시스템/프로그램 예시

## 22.1 모바일 앱

앱 안에 AI가 있다고 생각해봅니다.

앱이 이런 JSON을 읽을 수 있습니다.

```json
{
  "tone": "gentle",
  "short_answer": true,
  "use_examples": true
}
```

앱은 이걸 보고 다음처럼 판단할 수 있습니다.

- 말투는 부드럽게
- 답변은 짧게
- 예시는 포함하기

## 22.2 서버

웹사이트 뒤에는 서버가 있습니다.

사용자가 질문하면 서버가 AI에게 이런 정보를 줄 수 있습니다.

```json

  "user_level": "beginner",
  "language": "ko",
  "tone": "friendly"
}
```

서버는 이 데이터를 읽고 AI 답변 방식을 조절할 수 있습니다.

프로그램은 이 JSON을 읽고 어떤 agent를 사용할지 선택할 수 있습니다.

# 23. 왜 JSON은 시스템/프로그램에서 관리하기 쉬울까?

## 23.1 구조가 분명함

JSON은 항목과 값이 명확합니다.

```json
{
  "이름": "초코",
  "나이": 2
}
```

컴퓨터는 `이름`을 꺼내고 싶으면 정확히 꺼낼 수 있습니다.

## 23.2 코드에서 바로 읽기 쉬움

예를 들어 파이썬에서는 이런 식으로 값을 꺼낼 수 있습니다.

```python
data["이름"]
```

자바스크립트에서는 이런 식으로 꺼낼 수 있습니다.

```javascript
data.name
```

## 23.3 자동 처리에 좋음

JSON은 다음에 좋습니다.

- 저장하기
- 불러오기
- 서버로 보내기
- 앱에서 읽기
- 설정값으로 관리하기
- AI agent 옵션 관리하기

# 24. Markdown은 왜 시스템보다 사람에게 더 좋을까?

Markdown은 사람이 보기 좋은 문서입니다.

```markdown
# 역할
친절한 AI

- 말투: 부드럽게
- 길이: 짧게
```

사람은 이걸 쉽게 읽습니다.

하지만 컴퓨터가 정확히 `말투`만 꺼내려면 JSON보다 어렵습니다.

그래서 보통 이렇게 생각하면 됩니다.

- Markdown = 사람이 읽고 수정하기 좋음
- JSON = 컴퓨터가 읽고 처리하기 좋음

---

# 25. 최종 핵심 정리

## JSON

- 컴퓨터가 읽기 좋은 데이터 형식
- 앱, 서버, API, 설정값에 많이 사용
- 구조가 엄격함
- `true`, `false`, `null` 같은 실제 데이터 값을 사용함
- 시스템/프로그램이 자동 처리하기 좋음

## Markdown

- 사람이 읽기 좋은 문서 형식
- 제목, 목록, 표, 링크, 체크박스 등을 쉽게 표현함
- README, 설명서, 프롬프트, 메모에 많이 사용
- AI에게 지시문이나 페르소나를 설명할 때 편함

## AI Agent 페르소나

- 처음 설계할 때는 Markdown이 좋음
- 앱/서버에서 설정값으로 관리할 때는 JSON이 좋음
- 실전에서는 Markdown과 JSON을 같이 쓰면 좋음

---

# 26. 마지막 한 줄 요약

> JSON은 컴퓨터가 읽는 정리표이고, Markdown은 사람이 읽는 문서입니다. AI agent 페르소나는 Markdown으로 설명하고, 설정값은 JSON으로 관리하면 가장 좋습니다.
